# Plan: Refactor event system into a state machine

## Problem analysis

### Structural issues

1. **Implicit chains** â€” handlers call `dispatch()` on other events, creating hidden control flow. `Authenticated â†’ FetchAndSelectRepos â†’ SelectRepo â†’ FetchRepoFiles â†’ ReadRepoFilesContent` is a 5-deep chain that's invisible at the call site. Adding a step or changing order requires tracing through multiple files.

2. **No loading states** â€” the app has only `Initializing`/`Initialized` for `AppState` and a boolean `repoReadyAtom`, but can't represent "currently cloning" or "currently fetching repos." The UI shows nothing between login and files-ready.

3. **Invalid state combinations** â€” nothing prevents `authState === Authenticated` with `userAtom === null`, or `repoReadyAtom === true` with `filesAtom === {}`. The 8 independent atoms have 2^8 combinations; only a handful are valid.

4. **Fragile resets** â€” `handleLogout` manually resets 8 atoms. `handleSelectRepo` manually resets 4. Adding a new atom means remembering to update every reset path.

5. **Hard to expand** â€” adding a new feature (e.g. "syncing" state, "saving" state) means more atoms, more events, more chains, and hoping you got all the combinations right.

### Edge case bugs in the current system

6. **No error handling around git operations** â€” `handleFetchRepoFiles` calls `git.clone()` and `git.pull()` with no try/catch. If either throws (network error, auth failure, corrupt IndexedDB), the promise rejects unhandled. The `appStateAtom` may never reach `Initialized`, leaving the app stuck on the initializing screen forever.

7. **Data races on concurrent transitions** â€” `dispatch()` is async but has no concurrency control. If the user clicks "switch repo" while a clone is in progress, two transitions run simultaneously:
   - The in-flight `FetchRepoFiles` completes and sets `repoFilenamesAtom`/`filesAtom` for the _old_ repo.
   - The new `SelectRepo` wipes FS and starts a new clone.
   - Depending on timing, the old handler's atom writes can land _after_ the new handler's, corrupting state with stale data.

8. **`setSelectedRepo(null)` bypasses the machine** â€” in `app.tsx`, the "switch repo" button calls `setSelectedRepo(null)` directly on the Jotai atom, bypassing the event system entirely. This doesn't clear files, doesn't wipe FS, and leaves `repoReadyAtom === true` with stale files still rendered.

The new design also handles two unlikely but possible edge cases for free: partially corrupt IndexedDB (`.git` exists but objects are missing â€” pull throws with no recovery) and expired tokens mid-session (auth errors surface as cryptic failures with no logout button). These aren't primary motivations but fall out naturally from the error recovery design.

## Proposed solution: single discriminated-union state machine

Replace the scattered state atoms and event/dispatch system with a single state machine atom using a TypeScript discriminated union, with a `send()` function that enforces valid transitions and handles concurrency.

### State type

Each variant carries exactly the data available in that phase â€” no more, no less:

```typescript
type Repo = { owner: string; repo: string };

type AppMachine =
  | { phase: "initializing" }
  | { phase: "unauthenticated"; authError: string | null }
  | { phase: "fetchingRepos"; user: User }
  | { phase: "selectingRepo"; user: User; repos: string[] }
  | { phase: "cloningRepo"; user: User; repos: string[]; selectedRepo: Repo }
  | {
      phase: "loadingFiles";
      user: User;
      repos: string[];
      selectedRepo: Repo;
      filenames: string[];
      loaded: number;
    }
  | {
      phase: "ready";
      user: User;
      repos: string[];
      selectedRepo: Repo;
      filenames: string[];
      files: Record<string, TextFile | ImageFile>;
    }
  | { phase: "error"; message: string; user: User };
```

Key design decisions:

- `cloningRepo` and `loadingFiles` are separate phases â€” see "Performance edge cases" below for the rationale. Clone/pull is network-bound; file reading is I/O-bound. They have different bottleneck profiles and different progress semantics.
- `error` carries `user` so the UI can still show the username and offer logout. Error recovery can use the user context to retry.
- `authError` on `unauthenticated` is only set when coming from an OAuth error in the URL hash. Normal token expiry transitions to `unauthenticated` without an auth error (the user just sees the login button again).

### Transition function (`send`)

A single `send()` function replaces `dispatch()`. The transitions are:

```typescript
type Transition =
  | { type: "AUTHENTICATE"; user: User }
  | { type: "LOGOUT" }
  | { type: "SELECT_REPO"; owner: string; repo: string }
  | { type: "SWITCH_REPO" }
  | { type: "RETRY" };
```

Each transition handler is a single async function that does all work inline â€” no chaining:

```typescript
async function send(transition: Transition): Promise<void> {
  // 1. Validate the transition is allowed from the current phase
  // 2. Set machine to intermediate phase (e.g. fetchingRepos)
  // 3. Do async work (API calls, git ops) with try/catch
  // 4. Set machine to final phase (e.g. ready) or error phase
}
```

Example: `AUTHENTICATE` does the entire flow:

```
fetchingRepos â†’ fetch repos â†’ selectingRepo or cloningRepo â†’ clone â†’ loadingFiles â†’ read files â†’ ready
```

All in one function, with intermediate `store.set(machineAtom, ...)` calls between each step so the UI updates.

### Concurrency control

Add an **abort guard** to prevent data races from concurrent transitions:

```typescript
let activeController: AbortController | null = null;

async function send(transition: Transition): Promise<void> {
  // Abort any in-flight transition
  activeController?.abort();
  const controller = new AbortController();
  activeController = controller;

  // Before each async step, check if we've been superseded
  function checkAborted() {
    if (controller.signal.aborted) throw new AbortError();
  }

  try {
    // ... do work, calling checkAborted() between steps ...
  } catch (e) {
    if (e instanceof AbortError) return; // silently exit, a newer transition took over
    // handle real errors
  }
}
```

This solves the race condition: if the user clicks "switch repo" while a clone is in-flight, the old transition aborts silently and the new one takes over.

### Error recovery

The current system has no recovery paths. The state machine adds:

1. **Auth errors during async work** â€” if `fetchUserRepos()` returns null or git throws a 401/403, transition to `unauthenticated` (not `error`). The user sees the login button and can re-authenticate. The machine clears localStorage-persisted `userAtom` so there's no stale token on next load.

2. **Non-auth errors** â€” if git operations fail for other reasons (network, corrupt FS), transition to `error` phase. The error phase has a `user` field so the UI can show a retry button and a logout button. Retry wipes IndexedDB and re-enters from `fetchingRepos`.

3. **Corrupt IndexedDB recovery** â€” when `cloningRepo` detects `isInitialized() === true`, it tries `git.pull()`. If pull throws, it falls back to `wipeFs()` + `git.clone()`. This handles the partial-corruption case without user intervention.

### Persistence

`userAtom` and `selectedRepoAtom` remain as separate `atomWithStorage` atoms â€” they're the persistence layer. The `send()` function reads/writes them as side effects:

- `AUTHENTICATE` writes `userAtom`
- `SELECT_REPO` writes `selectedRepoAtom`
- `LOGOUT` clears both
- `init()` reads both to determine the starting transition

The machine state itself is **not** persisted. On page load, `init()` reads the cached atoms and replays the appropriate transition from scratch, which re-validates the token and re-fetches/clones. This means a hard refresh always gets fresh data â€” no stale state.

### Derived atoms

`noteListAtom` stays as a derived atom but reads from the machine state:

```typescript
const noteListAtom = atom<NoteListEntry[]>((get) => {
  const machine = get(machineAtom);
  if (machine.phase !== "ready") return [];
  // derive from machine.filenames and machine.files
});
```

`filesAtom` (used by `routes/note.tsx`) becomes a derived atom that reads from the machine:

```typescript
const filesAtom = atom((get) => {
  const machine = get(machineAtom);
  return machine.phase === "ready" ? machine.files : {};
});
```

This preserves the API for components that only need files, while guaranteeing they only see data when the app is actually ready.

### Component consumption

Components switch on `machine.phase` for type-safe rendering:

```typescript
function App() {
  const [machine] = useAtom(machineAtom);

  switch (machine.phase) {
    case "initializing":
      return <Loading />;

```

The "switch repo" button calls `send({ type: "SWITCH_REPO" })` instead of directly mutating an atom. This ensures FS cleanup and state consistency.

### Findings

**`errorAtom` is a dead-end.** `handleError` sets it, but no component reads it. The error is stored but never displayed. In the machine, the `error` phase will be explicitly rendered in `app.tsx`.

**`repoReadyAtom` is invisible.** It's set in `handleReadRepoFilesContent` but never read by any component. It serves no purpose. The `ready` phase replaces it.

**`selectedRepoAtom` is directly mutated by `app.tsx`.** The "switch repo" button calls `setSelectedRepo(null)` which bypasses the event system. This leaves `repoReadyAtom === true` and stale files in `filesAtom`. The machine eliminates this by requiring `send({ type: "SWITCH_REPO" })`.

**`userAtom` and `selectedRepoAtom` should become internal.** Currently components import and read these directly. In the machine, components get user/repo data from the machine state. The atoms should still exist for localStorage persistence but should NOT be exported for component use â€” they're implementation details of the persistence layer.

The total atom count drops from 10 writable + 1 derived to **3 writable** (machine + 2 persistence) **+ 2 derived** (files + noteList). The mutation surface drops from `dispatch()` + direct `store.set()` on any atom to a single `send()` function.

### Why not absorb `userAtom`/`selectedRepoAtom` into the machine?

They need to survive page reloads via `atomWithStorage`. The machine state is ephemeral â€” on load, `init()` reads the cached atoms and replays transitions from scratch to re-validate. If we persisted the entire machine state, we'd have to deal with schema versioning, stale phases, and deserialization of complex objects (Blobs in `files`). Keeping the persistence layer thin (just user credentials + repo selection) avoids all of that.

## Performance edge cases

### Large git repos

**Current behaviour.** `git.clone()` uses `depth: 1` and `singleBranch: true`, which limits downloaded objects. But a repo with many files or large binary assets will still produce a slow clone. After clone, `readRepoFiles()` walks the entire directory tree sequentially (`readdir` â†’ `lstat` per entry), then `handleReadRepoFilesContent()` fires `Promise.all()` across every file â€” all concurrent IndexedDB reads at once. For a 500-file repo, that's 500 parallel IDB reads plus holding all file contents in memory simultaneously.

**Problems:**

1. All file contents are held in memory simultaneously. A repo with hundreds of images (stored as Blobs) alongside thousands of text files creates unnecessary memory pressure during loading.
2. With no abort points during file reading, the user can't switch repos or cancel mid-load â€” they have to wait for the entire `Promise.all()` to finish.
3. No progress feedback â€” the user sees a blank spinner with no indication of how far along the load is.

**Machine design response:**

- **Separate `loadingFiles` phase** carries `filenames` (the full list from the tree walk) and `loaded` (count of files read so far). The UI can show "Loading files (42 / 156)..." instead of a blank spinner.
- **Batched file reading.** Instead of `Promise.all()` over all files, `send()` reads files in batches (e.g., 20 at a time). Between batches, it updates `loaded` on the machine atom and calls `checkAborted()`. This:
  - Caps memory pressure by only holding one batch of pending reads at a time.
  - Gives the abort guard a chance to cancel between batches (e.g., user clicks "switch repo").
  - Provides incremental progress updates to the UI without per-file re-renders.
- The batch size (20) is a reasonable default â€” small enough to limit concurrent memory, large enough to avoid excessive machine atom updates. It can be tuned later without changing the state type.

**Future optimisation: lazy file loading.** For very large repos, loading all file contents eagerly is wasteful â€” most notes won't be viewed in a session. A future iteration could make `ready` carry only `filenames`, with file content loaded on demand when a note is opened. This would require:

- `note.tsx` to trigger a lazy read (e.g., via a Jotai family atom keyed by path).
- `noteListAtom` to derive titles from filenames first (date-based, stem-based), then refine with content-derived titles as files load in the background.

This is out of scope for the initial refactor but the machine design supports it â€” `ready` already has a `filenames` field, and the `files` field could become a progressively-filled cache without changing the phase structure.

### Slow network

**Current behaviour.** `git.clone()`, `git.pull()`, `fetchUserRepos()`, and `fetchCurrentUser()` have no timeouts. On a slow or dead network:

- `fetchUserRepos()` hangs until the browser's built-in fetch timeout (which varies by browser and can be minutes).
- `git.clone()` and `git.pull()` use isomorphic-git's HTTP layer, which also hangs until the underlying fetch times out.
- The user sees "Initializing..." or a blank screen with no feedback and no way to cancel.

**Machine design response:**

- The abort guard gives the architecture a natural place to add cancellation later. A future enhancement could add a cancel button to `cloningRepo`/`loadingFiles` that calls `send({ type: "SWITCH_REPO" })` or `send({ type: "LOGOUT" })`, which aborts the in-flight work via the existing guard. Similarly, a `gitProgressAtom` driven by `onProgress` callbacks could provide fine-grained clone progress. Neither is in scope for this refactor â€” the existing `console.debug` progress callbacks are preserved as-is.
- **Timeout is not added.** Hard timeouts are problematic â€” a user on a 2G connection cloning a legitimate repo would be interrupted. If a timeout is needed later, `AbortSignal.timeout()` can be composed with the abort controller.

### Slow devices

**Current behaviour.** All IndexedDB operations (LightningFS) are async but can be slow on low-end hardware. The `readRepoFiles()` tree walk does sequential `readdir` + `lstat` calls â€” for a deep directory structure, this is many sequential IDB round-trips. The subsequent `Promise.all()` fires all file reads concurrently, potentially overwhelming IDB.

**Machine design response:**

- **Batched reads** (described above) limit concurrent memory pressure by processing files in groups of 20 rather than all at once.
- **Machine atom updates are coarse-grained.** The machine atom is only updated at batch boundaries (every 20 files), not per-file. This means React re-renders happen ~100 times for a 2000-file repo during loading, not 2000 times. Since each machine atom update creates a new object, minimising update frequency is important.
- **The tree walk (`readRepoFiles`) happens once, before file reading begins.** Its result (the `filenames` array) is set on the `loadingFiles` phase. This is a single atom update that gives the UI the total file count for progress display. The walk itself is sequential and relatively fast even on slow devices (directory listings are small IDB reads).
- **`noteListAtom` derivation is deferred.** It only runs when `phase === "ready"`, not during `loadingFiles`. This means the expensive `extractTitle()` calls (which run a unified/remark pipeline per `.md` file) don't happen until all files are loaded. On a slow device, this prevents the derivation from competing with file loading for CPU time.

### Summary of state type changes for performance

The `loadingFiles` phase is the key addition:

```typescript
| { phase: "loadingFiles"; user: User; repos: string[]; selectedRepo: Repo;
    filenames: string[]; loaded: number }
```

- `filenames` â€” full list from tree walk, set once at phase entry. Gives total count for progress UI.
- `loaded` â€” updated at batch boundaries. Progress = `loaded / filenames.length`.

A `gitProgressAtom` for fine-grained clone progress and a cancel button for long-running operations are natural future additions that the machine design supports, but are not part of this refactor.

## What this buys us

- **One place to understand app flow** â€” `send()` in `machine.ts` contains every transition. No hidden chains.
- **Type-safe state access** â€” if `phase === "ready"`, TypeScript guarantees `user`, `repos`, `selectedRepo`, `filenames`, and `files` all exist.
- **No data races** â€” the abort guard ensures only one transition runs at a time. Stale async work is silently cancelled.
- **No stale data** â€” machine state is never persisted. Hard refresh replays from localStorage cache, re-validates token, re-fetches everything.
- **Expired token recovery** â€” auth failures during any async step transition to `unauthenticated`, clearing the stale token and showing the login button.
- **Corrupt FS recovery** â€” failed pull falls back to wipe + clone. The user never gets stuck.
- **Atomic resets** â€” logout is one atom write: `{ phase: "unauthenticated", authError: null }`. No forgotten atoms.
- **Easy to expand** â€” adding "syncing" or "saving" is one variant + one transition case.
- **Visible loading states** â€” the UI can distinguish "fetching repos" from "cloning repository."
