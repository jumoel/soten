# Cache Markdown Parsing via Jotai Atom

## Problem

Markdown is parsed redundantly in two places:

1. **`note.tsx:40`** — `renderMarkdown(file.content)` runs the full unified pipeline (6 plugins, parse → GFM → frontmatter → remark-rehype → rehype-raw → stringify) inside an async component. Re-runs on every render of `Note`, even when content is unchanged.
2. **`store.ts:98`** — `extractTitle(content)` runs `titleParser.parse()` (remarkParse + remarkFrontmatter) for every `.md` file whenever `noteListAtom` re-derives.

## Scale Analysis: Thousands of Files

With a repo of N markdown files:

- **`extractTitle`**: Called N times synchronously on the main thread whenever `noteListAtom` re-derives. `noteListAtom` derives from `machineAtom`, so it recomputes any time `machineAtom` is set — including the single transition to `"ready"`. During the `"ready"` phase, `machineAtom` is stable (no edits exist yet), so this runs **once per repo load**. For thousands of files this is a one-time cost, not a per-render cost. `extractTitle` only calls `titleParser.parse()` (no transforms, no stringify) — it's the cheapest possible unified operation.

- **`renderMarkdown`**: Called once per render of the `Note` component, for **one file at a time** (the currently viewed note). This is O(1) relative to repo size, but it's the full 6-plugin pipeline and it re-runs on every re-render. This is where caching provides the most benefit per call.

**Conclusion**: The real inefficiency is `renderMarkdown` re-running on re-renders, not the bulk `extractTitle` at load time.

## Should the Pipelines Be Combined?

No. Combining them is counterproductive:

- **Different scale**: `extractTitle` runs for all N files (bulk, at load time). `renderMarkdown` runs for 1 file (on view). Combining them would mean either running the full heavy pipeline for all N files at load time (much worse), or still needing a separate light parser for the bulk case (no simplification).
- **Different sync requirements**: `extractTitle` is synchronous (needed for Jotai's synchronous derived `noteListAtom`). `renderMarkdown` is async. Combining them would force async title extraction, requiring changes to `noteListAtom` and `FrontPage`.
- **`extractTitle` is already minimal**: It only calls `.parse()` — no AST transforms, no serialization. There's nothing meaningful to save by avoiding it.

## Design: `renderedNoteAtom` Atom Family

Use Jotai's `atomFamily` to create a per-path derived atom that memoizes the rendered markdown for each viewed file.

### How it works

```ts
// store.ts
import { atomFamily } from "jotai/utils";

type RenderedNote = { frontmatter: Record<string, unknown> | null; html: string };

export const renderedNoteAtom = atomFamily((path: string) =>
  atom(async (get) => {
    const files = get(filesAtom);
    const file = files[path];
    if (!file || file.type !== "text") return null;
    return renderMarkdown(file.content);
  }),
);
```

- **Memoization**: Jotai derived atoms only recompute when dependencies change. `filesAtom` is reference-stable while `machineAtom` stays in `"ready"`. So after the first render, the atom returns the cached promise result.
- **Lazy**: Atoms are only created when a path is first accessed. Files the user never views are never parsed.
- **Suspense-compatible**: Async derived atoms integrate with the existing `<Suspense>` boundary in `NotePage`.
- **Automatic invalidation**: On repo switch or logout, `machineAtom` changes → `filesAtom` returns `emptyFiles` → all derived atoms recompute to `null`.

### Updated `note.tsx`

```tsx
function Note({ path }: { path: string }) {
  const [rendered] = useAtom(renderedNoteAtom(path));
  if (!rendered) return null;

  return (
    <>
      <Frontmatter data={rendered.frontmatter} />
      <div className="prose" dangerouslySetInnerHTML={{ __html: rendered.html }} />
    </>
  );
}
```

No longer `async`. No direct `renderMarkdown` call. The atom handles caching and async.

### What stays the same

- **`extractTitle`** — unchanged. It's sync, cheap, and runs once per repo load.
- **`noteListAtom`** — unchanged. Derives titles using `extractTitle` as before.
- **`renderMarkdown`** — unchanged signature. Just called from the atom instead of the component.
- **`markdown.ts`** — no changes needed.

## Implementation Steps

1. **`src/atoms/store.ts`**: Import `atomFamily` from `jotai/utils` and `renderMarkdown` from `../markdown`. Add `RenderedNote` type and `renderedNoteAtom` atom family.

2. **`src/atoms/globals.ts`**: Re-export `renderedNoteAtom`.

3. **`src/routes/note.tsx`**: Replace import of `renderMarkdown` with import of `renderedNoteAtom`. Remove `async` from `Note`. Use `useAtom(renderedNoteAtom(path))` instead of `await renderMarkdown(file.content)`. Remove `filesAtom` import (no longer needed here).

4. **Run quality checks**: `npm run lint && npm run types && npm run style && npm run build`.
